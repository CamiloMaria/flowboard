## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| guest→API | Guest JWT holders attempt mutation API calls on demo board |
| bot→services | Bots call BoardService/PresenceService directly (trusted, server-side) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | E (Elevation) | board.controller | mitigate | Guest role check on all mutation endpoints: `req.user.role === 'guest' && boardId === DEMO_BOARD_ID` → ForbiddenException. Guards prevent guests from modifying demo board data. |
| T-04-04 | T (Tampering) | demo.service | accept | Bots call services directly (no WebSocket). Only server code can trigger bot actions. No external input to bot choreography. |
| T-04-05 | D (DoS) | demo.service | mitigate | Grace period prevents rapid start/stop cycling. Guest count tracking ensures bots start once. Idempotent startBots() — calling twice is a no-op. |
| T-04-06 | S (Spoofing) | board.gateway | accept | Guest role comes from JWT (signed, server-verified). Cannot be spoofed without JWT secret. Existing ws-auth.middleware validates JWT on connect. |
</threat_model>
